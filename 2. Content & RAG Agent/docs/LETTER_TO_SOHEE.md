소희야 안녕! 나 다희야. 🙋‍♀️

RAG랑 콘텐츠 에이전트(2번) 쪽 이독성/난이도 독립변수 개편이랑 요약 기능 구현까지 다 완료해서 깃허브 `main` 브랜치에 안전하게 올려두었어! 

너가 연동할 때 헷갈리지 않게 변경 사항이랑 어떻게 가져다 쓰면 되는지 가이드를 정리해서 남겨놓을게.

## 1. 가독성(이독성) & 난이도 완전 독립변수화

원래 `난이도 = 100 - 가독성`이던 단순 대칭 관계를 끊고, 교수님 피드백을 반영해서 완전히 다른 독립변수로 쪼개놓았어.
* **이독성 (`readability_score`)**: 순수하게 문장 길이, 어절 복잡도, 국립국어원 기초어휘 기준 등 **문법적/형태적(표면적) 요인**으로만 계산해 (0~100점).
* **난이도 (`difficulty_score`)**: 텍스트 내부의 **고시 표준 전문 용어 밀도**를 기반으로 계산해 (0~100점).

### 💡 [핵심] 난이도 '지수 스케일링(Exponential Scaling)' 도입
단순히 전문 용어 개수 비율로 계산하니까 "양자역학" 같은 개념이 단 하나만 들어가도 독자가 느끼는 인지 장벽이 너무 낮게 평가되는 문제가 있더라고. 
그래서 사람이 느끼는 인지적 감각에 맞춘 **비선형 지수 공식**을 새로 넣었어:

$$\text{Difficulty Score} = 100 \times \left(1 - e^{-0.25 \times \text{전문용어밀도(\%)}}\right)$$

* 이제 "양자역학은..." 문장처럼 전문 용어가 단 하나만 박혀 있어도 난이도가 **73.4점**으로 강력하게 잡혀서 실제 체감 장벽이 제대로 반영될 거야!

## 2. 문단별 맞춤 요약 (`summary` 필드) 추가

너가 OX 퀴즈 만들 때 활용할 수 있게, **각 청크(문단)별로 독자 문해력 수준(1~5레벨)에 딱 맞춘 요약문을 생성하는 파이프라인**을 추가해 놓았어!
* 이제 `/api/content-reducer/reduce` 나 세션 생성 API를 호출하면 각 청크 딕셔너리 안에 **`summary`** 라는 필드가 추가로 들어갈 거야.
* **연동 방법**: 3번 백엔드에서 OX 퀴즈를 출제할 때, 원문 대신 이 **`summary`** 필드 값을 가져와서 `QuizGenerationRequest`의 `context`로 넘겨주면, 훨씬 매끄럽고 명확한 OX 퀴즈가 만들어질 거야!
* **비용 방어**: 교내 무료 플랫폼인 `SnowChat` (Gemini 2.5 Flash)을 써서 비용은 0원이고, 만약 API 서버에 장애가 나면 본문 앞부분을 잘라서 요약문으로 대체하는 안전장치(Fallback)도 꼼꼼하게 다 심어놓았으니까 크래시 날 걱정은 안 해도 돼!

## 3. 너(1번, 3번)가 수정/연동할 때 확인할 사항들

### 1번 Orchestrator (오케스트레이터)
* 기존의 E2E 테스트 코드(`test_content_e2e.py`)에 있던 **"가독성 + 난이도 = 100"** 강제 Assert 제약식은 내가 지우고 독립 0~100 범위 검증으로 바꿔놓았어.
* API 규격 호환성을 위해 `calculate_difficulty_score()` 함수에 **이중 가드(Polymorphic Guard)**를 탑재해서, 혹시 이전처럼 실수로 점수(float)를 넘겨도 `100 - score`로 안전하게 리턴되고, 텍스트(str)를 넘기면 독립된 난이도를 계산하도록 처리해 뒀어!

### 3번 Backend (백엔드)
* 세션 상태나 DB에 적재할 때 가독성 점수(`readability_score`)와 난이도 점수(`difficulty_score`)가 대칭이 아니니까, **개별 독립 칼럼**으로 온전히 보관해 줘.

### 4번 Frontend (프론트엔드)
* 화면에 가독성과 난이도를 다르게 표현하면 훨씬 예쁠 것 같아! (가독성 = "문장의 매끄러움", 난이도 = "주제의 전문성/인지 부하" 식으로 다중 차트나 뇌 아이콘 등으로 다채롭게 시각화 제안)
